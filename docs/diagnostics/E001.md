# E001 — Type mismatch

The type of an expression doesn't match what the surrounding context expects.

## Common cases

```almide
let x: Int = "hello"             // expected Int, got String
fn f() -> Int = "abc"            // body type doesn't match return type
list.map(xs, (x) => x + "!")     // closure return type conflicts with list element
```

## Specialized sub-patterns (with `try:` snippets)

### Unit leak: fn body ends with a `let` binding

```almide
fn sum_digits(n: Int) -> Int = {
    let abs_n = int.abs(n)           // ← `let` returns Unit, not Int
}
```

Diagnostic:

```
error[E001]: type mismatch in fn 'sum_digits': expected Int but got Unit
  try:
      // fn body ends with `let abs_n = ...` (a statement, returns Unit).
      // Add `abs_n` as the trailing expression so the fn returns Int:
      //
      //   let abs_n = <computation>
      //   abs_n                         // <-- add this line
```

The compiler extracts the real binding name from the AST and cites it in
the snippet. Fix: add the name as a trailing expression.

### Unit leak: if-arm is an assignment

```almide
fn clamp(x: Int, lo: Int) -> Int = {
    var val = x
    if val < lo then { val = lo } else val    // ← then-arm is Unit
}
```

Diagnostic:

```
error[E001]: type mismatch in if branches: expected Unit but got Int
  try:
      // the then-arm is `val = ...` (assignment, returns Unit).
      // Rewrite as a rebinding of `val`:
      //
      //   let new_val = if cond then <new-value-for-val> else val
```

The real variable name (`val`) is cited. Fix: use `let new_x = if cond ... else ...`.

## Fix (general)

- Adjust the literal/value to match the expected type.
- Change the explicit type annotation to match the actual expression.
- Convert with stdlib helpers: `int.to_string`, `int.parse(s)?`,
  `float.from_int`, `float.to_int` (truncating), etc.
- **If the "got Unit" half of a mismatch is suspicious**, inspect
  whether a trailing `let x = ...` or an `x = y` assignment is in a
  position expected to yield a value.
