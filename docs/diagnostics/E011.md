# E011 — Mutable var mutated inside closure in pure fn

A `var` declared in a pure function (`fn`) is being reassigned from
inside a lambda / closure nested in that function. Closures capture
by value in Almide's pure semantics; mutating a captured `var` from
inside a closure would require a hidden shared cell, which breaks
referential transparency.

## Common cases

```almide
fn count_evens(xs: List[Int]) -> Int = {
    var count = 0
    list.each(xs, (x) => {
        if x % 2 == 0 then count = count + 1    // E011: closure mutation
    })
    count
}
```

## Diagnostic

```
error[E011]: mutable variable 'count' is mutated inside a closure in a pure function — use effect fn instead
  hint: Move the mutation out of the closure, or mark the enclosing function as `effect fn`
```

## Fix — prefer functional form (in pure fn)

For counting / accumulating, use `list.fold` / `list.count`:

```almide
fn count_evens(xs: List[Int]) -> Int =
    list.count(xs, (x) => x % 2 == 0)
```

For transformations, use `list.map`:

```almide
// ✗
fn doubles(xs: List[Int]) -> List[Int] = {
    var out: List[Int] = []
    list.each(xs, (x) => { out = out + [x * 2] })
    out
}

// ✓
fn doubles(xs: List[Int]) -> List[Int] = xs |> list.map((x) => x * 2)
```

## Fix — mark as `effect fn` (when mutation is genuine)

If the mutation really is effectful (e.g. accumulates I/O results),
the enclosing function should be `effect fn`:

```almide
effect fn gather(xs: List[String]) -> Result[List[String], String] = {
    var out: List[String] = []
    for x in xs {                     // `for` is allowed; it's not a closure
        let text = fs.read_text(x)!
        out = out + [text]
    }
    ok(out)
}
```

`for` / `while` loops are NOT closures — they can mutate the outer
`var` without E011. Only `lambda` (`(x) => ...`) captures trigger the
check.

## Why

Pure fns must be referentially transparent — calling `f(x)` twice
must produce the same value. If a closure could mutate outer state,
that invariant breaks. Almide catches this statically and asks the
author to either refactor to a functional primitive (which is usually
shorter and LLM-friendlier anyway) or escalate to `effect fn`.
