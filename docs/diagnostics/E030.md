# E030 — type has no ordering

`list.sort`, `list.min`, `list.max`, and `list.sort_by`'s key require an
ORDERABLE element — the native runtime's `T: Ord` bound. A bare `Float`
element is fine (it routes to the IEEE-754 totalOrder `_float` twins), but a
`Map`, `Set`, or function element — or a `Float` nested inside a compound
element — has no order. Previously `check` accepted these while native rustc
rejected the monomorph (`AlmideMap: Ord is not satisfied` — the
check-vs-build acceptance-parity gap, differential-fuzz seed 20260718 index
629).

```almide
fn main() -> Unit = {
  let xs: List[Map[String, Int]] = [["a": 1]]
  let s = list.sort(xs)                       // ← E030: Map has no ordering
  println("${list.len(s)}")
}
```

## Fix

Sort by an explicit orderable key instead:

```almide
fn main() -> Unit = {
  let xs: List[Map[String, Int]] = [["a": 1], ["b": 2]]
  let s = list.sort_by(xs, (m) => map.len(m))   // Int key — orderable
  println("${list.len(s)}")
}
```

Orderable: `Int` (all widths), `Bool`, `String`, bare `Float`, and
lists/tuples/records built only from those. Not orderable: `Map`, `Set`,
functions, and `Float` inside a compound element (compare via an explicit
key).
