# E006 — Effect isolation: pure fn calls effect fn

A non-effect function (`fn`) tried to call an effect function
(`effect fn`). Almide tracks effects in the type system; pure fns
cannot depend on effectful ones.

## Common cases

```almide
effect fn read_text(path: String) -> String = ...

fn pure_fn() -> String = read_text("a.txt")    // E006: pure calls effect
```

## Diagnostic (with secondary note)

```
error[E006]: cannot call effect function 'read_text' from a pure function
  --> main.almd:3:25
  hint: Mark the calling function as `effect fn`
  note: 'read_text' declared as effect fn here
        main.almd:1:1
```

The secondary `note` points to the effect fn's declaration so the user
can see why the callee is restricted.

## Fix

- Convert the caller to `effect fn`:
  ```almide
  effect fn pure_fn() -> String = read_text("a.txt")
  ```
- Or push the effect call up to a higher-level `effect fn` and pass
  the already-computed value down:
  ```almide
  effect fn setup() -> String = read_text("a.txt")

  fn use_text(text: String) -> String = text        // stays pure
  ```
- If the "effect fn" call is actually a mistake (e.g. called
  `fs.read_text` when you meant a pure transformation), fix the call.

## Why

`effect fn` values carry `Result<T, String>` at the Rust level and
have implicit `?` propagation (`expr!`). Letting a `fn` invoke an
`effect fn` would silently lose error tracking and make reasoning
about effects impossible for both humans and LLMs. The boundary is
enforced statically.
