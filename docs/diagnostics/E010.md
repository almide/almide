# E010 — Non-exhaustive match

A `match` expression doesn't cover every possible case of its subject.

## Common cases

```almide
type Color = | Red | Green | Blue

fn name(c: Color) -> String =
    match c {
        Red   => "red",
        Green => "green",
        // E010: missing Blue
    }
```

```almide
let o: Option[Int] = some(5)
match o {
    some(n) => n,
    // E010: missing `none`
}
```

## Diagnostic

```
error[E010]: non-exhaustive match: missing `Blue`
  --> main.almd:4:5
  hint: add an arm for `Blue`, or use `_ => ...` to cover the rest
  in match
```

The diagnostic lists the specific missing constructor(s) by name.

## Fix

- Add the missing arm(s) explicitly:
  ```almide
  match c {
      Red   => "red",
      Green => "green",
      Blue  => "blue",          // added
  }
  ```
- Use a wildcard `_` as the final arm to cover all remaining cases:
  ```almide
  match c {
      Red => "red",
      _   => "other",
  }
  ```
- For `Option[T]`: `some(x) => ...` + `none => ...`
- For `Result[T, E]`: `ok(x) => ...` + `err(e) => ...`
- For `List[T]`: `[] => ...` + `_ => ...` (Almide has no cons-pattern;
  `_` + `list.first/list.drop` inside the arm is the idiomatic form —
  see E014-related hints on rest-patterns).

## Why exhaustive

Exhaustiveness catches refactoring mistakes: if you add a new variant
to `Color`, every `match` that doesn't use `_` will now fail to
compile, forcing you to decide how to handle the new case. This is
what "modification survival rate" means in practice — code that
doesn't silently ignore new values.
