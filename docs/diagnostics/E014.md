# E014 — Unreachable match arm

A `match` arm is statically covered by a preceding arm, so it can
never fire at runtime. Almide flags this as an error (not a warning)
because an unreachable arm is almost always a generation mistake —
the author mis-encoded an earlier pattern's reach and assumed the
later arm would catch a case it never sees.

## Common cases

```almide
type Shape = | Circle | Square | Triangle

fn area(s: Shape) -> Int =
    match s {
        _ => 0,
        Circle => 1,   // E014: `_` above already matches Circle
    }
```

```almide
match n {
    x if x > 0 => "positive",
    5 => "five",   // E014: covered by `x if x > 0` since 5 > 0
    _ => "other",
}
```

```almide
type Tree = | Leaf | Node(Tree, Tree)

fn size(t: Tree) -> Int =
    match t {
        Leaf => 0,
        Node(_, _) => 1,
        Node(Leaf, _) => 2,   // E014: earlier `Node(_, _)` catches this
    }
```

## Fix

Either delete the unreachable arm, or tighten the earlier arm so the
later one is reachable:

```almide
// Option A — drop the dead arm
match s {
    _ => 0,
}

// Option B — make the earlier arm more specific
match s {
    Circle => 1,
    _ => 0,
}
```

## Guarded arms

Guarded arms (`pat if cond => ...`) neither shadow later arms nor get
shadowed — the guard can fail at runtime, so the reachability
analysis treats them as "always possibly reachable" in both
directions.

```almide
match n {
    x if x > 0 => "positive",
    x if x < 0 => "negative",
    _ => "zero",     // OK: guarded arms above don't cover 0
}
```

## Why not a warning?

LLMs treat warnings as noise, and the dojo MSR shows unreachable
arms are almost exclusively the signal of a pattern-encoding mistake
(the author thought `Leaf` meant something different from what it
matches). Surfacing at error level keeps the fix in the first
feedback loop.
