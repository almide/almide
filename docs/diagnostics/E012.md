# E012 — Duplicate definition

Two declarations share the same name within a module (or two test
blocks share the same name within a file). Each name must be unique
in its scope.

## Common cases

```almide
fn greet(name: String) -> String = "hi ${name}"
fn greet(who: String) -> String = "hello ${who}"    // E012: duplicate fn
```

```almide
test "basic" { assert(1 == 1) }
test "basic" { assert(2 == 2) }                     // E012: duplicate test
```

## Diagnostic

```
error[E012]: function 'greet' is defined twice in this module
  hint: Rename one of the definitions, or remove the earlier one. Almide requires each function name to be unique within a module.
```

The secondary `note` points to the first definition so both sites are
visible.

## Fix

- Rename one of the definitions to reflect what it actually does:
  ```almide
  fn greet(name: String) -> String = "hi ${name}"
  fn greet_formal(who: String) -> String = "hello ${who}"
  ```
- Or delete the older / stale one if one of them is leftover code.
- For tests: combine the two into a single `test` block with both
  assertions, or name them to reflect different cases:
  ```almide
  test "basic — 1 equals 1" { assert(1 == 1) }
  test "basic — 2 equals 2" { assert(2 == 2) }
  ```

## Why no overloading

Almide rejects name overloading on purpose. Overload resolution is a
major source of LLM confusion in other languages: the same source
line can resolve to different implementations depending on argument
types, imports, and resolution rules. With no overloading, every call
site is unambiguous — `greet(x)` always means the single `greet` in
scope.

If you need two related behaviors, give them different names
(`greet_en`, `greet_formal`, etc.) or take the distinguishing
parameter explicitly.
