# E021 — Invalid record construction or pattern

A record (or record-payload variant case) was constructed or matched with the
wrong **shape**. Records are built from **named fields**; tuple cases are built
from **positional arguments** — mixing the two, or naming a field that does not
exist, silently corrupted the value before this check existed (the arguments
were dropped on the floor and the targets disagreed on what happened next).

`Cfg { name: "x" }` (brace) is the canonical spelling. `Cfg(name: "x")` (paren,
named-argument call syntax) is an accepted **alias**, normalized to exactly
the same construction — but only for construction and only for a plain
record type. A variant's record-payload case (`Circle { radius: Float }` as
one arm of `type Shape = Circle { radius: Float } | ...`) only ever matches
as a brace pattern, never `Circle(radius)` — see the last example below.

## Common cases

Positional arguments on a record:

```almide
type Cfg = { name: String }
let c = Cfg("x")          // E021: 'Cfg' takes named fields, not positional arguments
let ok1 = Cfg(name: "x")  // ✓ normalized to the brace form
let ok2 = Cfg { name: "x" } // ✓
```

Unknown / duplicate / missing fields (checked for both spellings):

```almide
type User = { name: String, age: Int = 0 }
let a = User { nmae: "x" }            // E021: 'User' has no field 'nmae'
let b = User { name: "x", name: "y" } // E021: field 'name' given more than once
let c = User { }                      // E021: missing field 'name' ('age' has a default — fine)
```

Named arguments on a tuple case, and brace construction of a tuple case:

```almide
type W = Wrap(Int) | E
let w = Wrap(value: 1)   // E021: constructor 'Wrap' takes positional arguments
let v = Wrap { x: 1 }    // E021: case 'Wrap' does not take named fields
```

Paren pattern on a record-payload case:

```almide
type Tag = | SetEmotion { name: String } | Other
fn f(t: Tag) -> String = match t {
  SetEmotion(_) => "s",       // E021: use a record pattern
  _ => "o",
}
// ✓ write: SetEmotion { .. }  (or bind fields: SetEmotion { name })
```

## Why this is an error

Positional binding to named fields would silently change meaning whenever a
field is inserted or reordered — the exact class of edit a record type exists
to survive. Named-field construction is order-independent and validated
against the declaration, so a typo or a missing required field is caught here
instead of leaking a generated-Rust error (`E0560`/`E0063`) or a wasm trap.
