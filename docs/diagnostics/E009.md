# E009 — Reassignment to immutable binding

`let x = ...` creates an immutable binding. Use `var` for mutable bindings.

## Common cases

```almide
let count = 0
count = count + 1            // E009: 'count' is immutable
```

## `try:` snippet

```
error[E009]: cannot reassign immutable binding 'count'
  hint: Use 'var count = ...' instead of 'let count = ...' to declare a mutable variable
  try:
      // let count = ...  →  var count = ...
      var count = <initial value>
```

The compiler extracts the real name from the AST. When the reassigned
binding is a **function parameter** (immutable), the snippet shifts:

```
  hint: 'count' is a function parameter (immutable). Use a local copy: var count_ = count
  try:
      // 'count' is a parameter — make a mutable copy:
      var count_ = count
      // ...then reassign count_ instead of count
```

## Fix

- Use `var` for mutability (prefer if the mutation is actually the
  design intent):
  ```almide
  var count = 0
  count = count + 1          // OK
  ```
- Prefer functional style: produce a new value, keep the old name
  immutable:
  ```almide
  let count = 0
  let count2 = count + 1
  ```
- For parameters specifically, take a local copy:
  ```almide
  fn step(x: Int) -> Int = {
      var x_ = x
      x_ = x_ + 1
      x_
  }
  ```

## Why immutable by default

Immutability maximizes LLM modification survival: a reader can trust
that `x` means the same value throughout a scope. Explicit `var`
signals intent to mutate — which usually co-occurs with `effect fn`
(side-effectful code) or a localized accumulator pattern.

For pure functions that feel like they need mutation (loops), prefer
recursion:

```almide
fn count_up(n: Int) -> Int =
    if n == 0 then 0
    else 1 + count_up(n - 1)
```
