# E008 — `fan` block captures mutable variable

A `fan { ... }` block tried to capture a `var` binding from the
enclosing scope. Fan branches may run concurrently, so reading/writing
shared mutable state would race.

## Common cases

```almide
effect fn broken() -> Int = {
    var count = 0
    fan {
        count = count + 1         // E008: mutable capture in fan
        count = count + 2
    }
    count
}
```

## Diagnostic

```
error[E008]: cannot capture mutable variable 'count' inside fan block
  hint: Use a `let` binding instead of `var` for values shared across fan expressions
```

## Fix

- Compute each branch independently, then aggregate the results
  after the fan block:
  ```almide
  effect fn fixed() -> Int = {
      let (a, b) = fan {
          compute_a()
          compute_b()
      }
      a + b
  }
  ```
- For accumulation patterns, use `fan { ... }` to produce a list of
  results and then `list.sum` / `list.fold` over it.

## Why

Concurrent writes to the same `var` produce undefined-order updates.
Almide rejects the pattern at the checker rather than emitting racy
code. `let` (immutable) bindings are always safe to read from fan
branches — only `var` is restricted.
