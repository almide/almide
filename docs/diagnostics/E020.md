# E020 — Conflicting type declarations

Two **distinct** types share the same bare name with **different
structures**. Type identity in Almide is by bare name through linking and
code generation, so the second declaration silently shadows the first — and
any function that used the shadowed type then fails with a cryptic
generated-Rust error (`E0560`/`E0609`: "no field …"). The checker reports the
collision at the source instead.

This most often happens **cross-package**: a local type collides with a
dependency's same-named type (e.g. a local `Persona` and `fizz_persona`'s
`Persona`). It also catches a duplicate type declaration within one file. A
structurally-identical re-declaration (the same type reached through two import
paths — the diamond case) is **not** flagged.

## Common cases

```almide
type Config = { name: String }
type Config = { host: String, port: Int }   // E020: 'Config' declared more than once

fn main() -> Unit = println("ok")
```

Cross-package — a dependency `dep` also declares `Config`:

```almide
import dep
type Config = { name: String }              // collides with dep's `Config`
```

## Fix

Rename one of the types so the bare name is unique. Types are not yet
namespaced per package, so renaming is the way to keep them distinct.

```almide
type AppConfig = { name: String }
type DepConfig = { host: String, port: Int }

fn main() -> Unit = println("ok")
```
