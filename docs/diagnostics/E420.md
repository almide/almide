# E420 — Function visibility violation

A function call attempted to reach a function that isn't visible from
the call site due to module visibility (it's declared `private`,
module-private, or lives in a submodule whose surface doesn't export
it).

Note: this code number is outside the usual `E001..E013` range and is
a candidate for renumbering to `E014` in a future release (see
`crates/almide-frontend/src/check/infer.rs`). The number is stable
until then — check your diagnostics for `E420` verbatim.

## Common cases

```almide
// my_module.almd
fn public_helper() -> Int = 42
fn _private_helper() -> Int = 0         // underscore-prefixed = private

// main.almd
import self.my_module
fn main() -> Unit = {
    let n = my_module._private_helper()    // E420
    println(int.to_string(n))
}
```

## Diagnostic

```
error[E420]: function 'my_module._private_helper' is not accessible
  hint: '_private_helper' is declared as `private` (module-internal only)
```

The `hint:` line names the visibility scope the callee actually has.

## Fix

- If the function should be part of the public surface, rename it
  without the underscore prefix (or remove the `private` annotation):
  ```almide
  fn public_helper() -> Int = 0     // now visible from outside
  ```
- If it's intentionally internal, add a thin public wrapper:
  ```almide
  fn public_entry() -> Int = _private_helper()     // public shim
  ```
- If the caller is in the same module and the callee still fails,
  check that you're not re-importing via an alias that breaks the
  private scope.

## Why visibility at all

Module-internal helpers should be refactorable without rippling
through unrelated callers. Almide's visibility system provides the
guarantee: underscore-prefixed names are stable within a module but
not across module boundaries.
