# E013 — Field access on a non-record / missing field

A `.field` access targeted a value whose type doesn't declare that
field (or isn't a record at all).

## Common cases

```almide
type Point = { x: Int, y: Int }

let p = Point { x: 1, y: 2 }
let z = p.z                      // E013: no field `z` on Point

let n: Int = 42
let bad = n.value                // E013: Int has no fields
```

## Diagnostic

```
error[E013]: no field `z` on type `Point`
  --> main.almd:4:9
  hint: Available fields on `Point`: x, y. Check spelling or verify the record type.
  in field access .z
```

When the type IS a record, the diagnostic lists its fields. For
non-records (Int, List, etc.), it notes that method-style access via
`.foo()` is not supported — use `module.foo(value)` instead (see
E002).

## Fix

- Spelling: check the declared field name(s). Records are case-sensitive.
- Wrong record: you may be accessing a field that exists on a
  different type. Verify `p`'s declared type matches the fields you
  expect.
- Method-style mistake: `xs.head` / `s.length` aren't fields, they're
  cross-language idiom imports. Rewrite as `list.first(xs)` /
  `string.len(s)`. `almide fix` doesn't auto-apply this yet (needs
  type info); see the E002 hint table.

## Why

Field access only applies to record types. Almide has no "method-call
syntax" that desugars `x.foo` into a function call on non-records —
this is a deliberate simplification to make resolution local and
unambiguous. The only cases where `.<name>` is valid are:

- Field access on records
- Tuple indexing (`.0`, `.1`, ...) with integer names
