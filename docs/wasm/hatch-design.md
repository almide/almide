# hatch — Almide WASM Agent MCP Bridge

Standalone tool that loads an Almide-compiled WASM binary and exposes it as an MCP server.

## Why standalone

MCP is a protocol, not a language feature. If MCP is superseded (A2A, WebMCP, or something new), hatch is replaced — the compiler and language are untouched.

## What hatch does

```
manifest.json  ──→  hatch  ──→  MCP Server (JSON-RPC 2.0 / stdio)
agent.wasm     ──→  hatch  ──→  wasmtime (WASM runtime)
```

1. **Read manifest.json** → generate MCP tool definitions (name, description, JSON Schema params)
2. **Load agent.wasm** into wasmtime → apply WASI capabilities (`--dir`, `--env`)
3. **MCP protocol loop** → `initialize` → `tools/list` → `tools/call` → dispatch to WASM
4. **Invoke WASM function** → pass args via stdin/memory → read result from stdout/memory

## What hatch does NOT do

- WIT parsing (manifest.json is enough)
- Component Model (core module direct execution)
- A2A protocol (separate tool if needed)
- HTTP transport (stdio only for v1; Streamable HTTP is future)
- Tool discovery logic (manifest declares everything at build time)

## manifest.json

Generated by `almide build --target wasm`. Contains everything hatch needs.

```json
{
  "name": "code-review-agent",
  "version": "0.1.0",
  "capabilities": ["FS.read", "IO.stdin", "IO.stdout"],
  "tools": [
    {
      "name": "read_file",
      "description": "Read file contents at the given path",
      "inputSchema": {
        "type": "object",
        "properties": {
          "path": { "type": "string", "description": "File path to read" }
        },
        "required": ["path"]
      }
    },
    {
      "name": "list_dir",
      "description": "List directory contents",
      "inputSchema": {
        "type": "object",
        "properties": {
          "path": { "type": "string", "description": "Directory path" }
        },
        "required": ["path"]
      }
    }
  ],
  "wasi_imports": ["fd_read", "fd_write", "fd_close", "path_open", "path_filestat_get"],
  "entry": "_start"
}
```

## CLI

```bash
# Serve: start MCP server on stdio
hatch serve agent.wasm
hatch serve agent.wasm --dir /workspace --dir /data
hatch serve agent.wasm --env API_KEY=xxx

# Inspect: show manifest without starting server
hatch inspect agent.wasm

# Validate: check manifest against WASM binary
hatch validate agent.wasm
```

## MCP Protocol Implementation

### initialize

```json
← {"jsonrpc":"2.0","method":"initialize","id":1,"params":{"capabilities":{}}}
→ {"jsonrpc":"2.0","id":1,"result":{"serverInfo":{"name":"hatch","version":"0.1.0"},"capabilities":{"tools":{}}}}
```

### tools/list

```json
← {"jsonrpc":"2.0","method":"tools/list","id":2}
→ {"jsonrpc":"2.0","id":2,"result":{"tools":[...from manifest.json...]}}
```

### tools/call

```json
← {"jsonrpc":"2.0","method":"tools/call","id":3,"params":{"name":"read_file","arguments":{"path":"main.py"}}}
```

hatch receives → serializes arguments to JSON → writes to WASM agent stdin → agent processes → reads stdout → returns:

```json
→ {"jsonrpc":"2.0","id":3,"result":{"content":[{"type":"text","text":"def main():\n    ..."}]}}
```

## Claude Code Integration

### Model A: Additive (MCP tool)

```json
// .claude/.mcp.json
{
  "mcpServers": {
    "hatch": {
      "type": "stdio",
      "command": "hatch",
      "args": ["serve", "agent.wasm", "--dir", "/workspace"]
    }
  }
}
```

Claude uses native tools AND hatch tools. Developer chooses.

### Model B: Substitutive (full harness)

```json
// .claude/settings.json
{
  "permissions": {
    "deny": ["Bash", "Edit", "Write"],
    "allow": ["Read", "Glob", "Grep", "mcp__hatch__*"]
  }
}
```

```json
// .claude/.mcp.json
{
  "mcpServers": {
    "hatch": {
      "type": "stdio",
      "command": "hatch",
      "args": ["serve", "agent.wasm", "--dir", "/workspace"]
    }
  }
}
```

Claude can ONLY operate through hatch. All writes go through capability-checked WASM.

## Implementation (Rust)

Dependencies:
- `wasmtime` — WASM runtime
- `serde_json` — JSON-RPC parsing
- `clap` — CLI

Core is ~500 lines:
- JSON-RPC stdio reader/writer (100 lines)
- MCP protocol handler: initialize, tools/list, tools/call (150 lines)
- WASM loader + WASI config (100 lines)
- Tool dispatch: JSON args → WASM stdin, WASM stdout → JSON result (100 lines)
- CLI (50 lines)

No framework dependency. No async runtime. Blocking stdio is correct for MCP stdio transport.

## Future

- **Streamable HTTP transport** — remote hatch server (MCP spec 2026-03-26)
- **A2A Agent Card** — `hatch a2a agent.wasm` generates `/.well-known/agent.json`
- **OCI distribution** — `hatch push agent.wasm ghcr.io/org/agent:latest`
- **Multi-agent** — `hatch compose agents.toml` runs multiple agents with restricted capabilities
