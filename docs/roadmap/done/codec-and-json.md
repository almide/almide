# Codec Protocol & JSON

## Vision

Trustworthy structured data boundaries for humans, programs, and models.

Rather than a JSON-specific API, design a format-agnostic **Codec protocol** with JSON as its first implementation.
Unify types, schemas, local updates, and repair into a standard language experience for data boundaries that LLMs can't easily break.

Core design philosophy: **JSON is a "conversion target," not a "writing target."** Don't let LLMs write JSON strings directly; have them write typed records and encode. Give LLMs records to write, and if needed, pass them a JSON Schema-compatible schema via `json.describe[T]()`.

```almide
// The right way — hard for LLMs to break
type Person = { name: String, age: Int } deriving Codec

let alice = Person { name: "Alice", age: 30 }
let text = json.encode_to_string(alice)           // Person -> String
let bob = json.decode_from_string[Person](text)?   // String -> Person
```

## Design Principles

1. **typed record → encode is the primary path** — LLMs write typed records. They don't write JSON strings
2. **derive is the primary path** — day-to-day use completes with `deriving Codec`. manual is for exceptions
3. **No field annotations** — handled by type-level configuration + manual implementation. Avoids annotation DSL explosion. No field annotations in initial stage. May consider minimal field-level codec override in the future if needed
4. **Json as first-class** — separate syntax (parse/stringify) from semantics (encode/decode)
5. **missing/null/default are explicit** — naturally resolved with Option + field defaults
6. **Errors are structured + repairable** — path + kind + suggested fix
7. **Local operations > full regeneration** — LLMs are more stable with path-based surgical edits

## Architecture

The Codec protocol is a format-agnostic abstraction; Json / parse / stringify / JsonOptions are JSON backend specifications.

- **Codec core**: derive, encode/decode concepts, defaults, naming, ADT repr
- **JSON backend**: Json, parse/stringify, JsonOptions, path API, repair rules

```
Text ──parse──▶ Json ──decode[T]──▶ T
T ──encode[T]──▶ Json ──stringify──▶ Text

Local operations:
Json ──set/remove/update(path)──▶ Json

Repair:
Json ──repair[T]──▶ RepairResult[T]

Convenience API:
T ──encode_to_string──▶ Text
Text ──decode_from_string[T]──▶ T
```

### Codec abstraction (internally separated)

"Codec" here is not a language-level trait/protocol, but a **codegen convention (naming convention + deriving directive)**.
This is a different concept from the container protocols (Mappable, etc.) in type-system.md.

```almide
// Conceptual definition (function groups generated by codegen)
// Encodable: T -> Json (or generic Encoder)
// Decodable: Json (or generic Decoder) -> T
// Codec = Encodable + Decodable
```

Internally separated to support cases where only encode or only decode is needed.
The user-facing surface recommends `deriving Codec`. If a trait-equivalent mechanism is added to the type system in the future, there may be a migration from codegen conventions to that.

## LLM Design Rationale

### 5 patterns where LLMs break JSON

1. **Syntax errors** — missing commas, forgotten quotes, unmatched `}`, broken escapes
2. **Key name errors** — typos, mixed snake/camel case
3. **Type errors** — numbers as strings, mixing null and missing
4. **Partial update errors** — regenerating everything and deleting existing fields
5. **Ambiguous failures** — unclear what's wrong, retrying breaks something else

### Almide's solutions

| Pattern | Solution |
|---------|----------|
| Syntax errors | Don't let them write JSON strings. typed record → encode |
| Key name errors | Automatic via `deriving Codec`. Bulk conversion via naming strategy |
| Type errors | Compile-time type checking. Record field types are guaranteed |
| Partial update errors | Path-based local operation API (`json.set_path`, `json.remove_path`) |
| Ambiguous failures | Structured errors (path + kind) + repair API |

### Choosing between 3 primary paths

```almide
// 1. Primary path: typed record → encode (LLMs write this)
let p = Person { name: "Alice", age: 30 }
let j = json.encode(p)

// 2. Fallback: Json builder (for dynamic JSON)
let j = json.object([("name", json.s("Alice")), ("age", json.i(30))])

// 3. Local operations: path API (for mutating existing JSON)
let j2 = json.set_path(j, Field(Field(Root, "user"), "name"), json.s("Bob"))
```

## Phase 1: JSON Builder + Path API (stdlib additions only)

### Manual Json construction

```almide
let person = json.object([
  ("name",    json.s("Alice")),
  ("age",     json.i(30)),
  ("active",  json.b(true)),
  ("address", json.object([
    ("city",  json.s("Tokyo"))
  ])),
  ("tags",    json.array([json.s("dev")])),
  ("notes",   json.null())
])
```

### Builder API

```
json.object     : (List[(String, Json)]) -> Json
json.array      : (List[Json]) -> Json
json.s          : (String) -> Json
json.i          : (Int) -> Json
json.f          : (Float) -> Json
json.b          : (Bool) -> Json
json.null       : () -> Json
```

### Parse / stringify

```
json.parse           : (String) -> Result[Json, String]
json.stringify       : (Json) -> String
json.stringify_pretty : (Json) -> String
```

### Access

In Phase 1, `DecodeError` is not yet implemented, so access APIs are `Option`-based.
Typed getters (`get_field` family) will be added after Phase 4's `DecodeError` introduction.

```
// object field access (raw)
json.get        : (Json, String) -> Option[Json]

// value extraction (Json → typed value)
json.as_string  : (Json) -> Option[String]
json.as_int     : (Json) -> Option[Int]
json.as_float   : (Json) -> Option[Float]
json.as_bool    : (Json) -> Option[Bool]
json.as_array   : (Json) -> Option[List[Json]]

// structure
json.keys       : (Json) -> List[String]
```

Use `json.get` and `json.as_*` in composition: `json.get(j, "name") |> json.as_string`.
From Phase 4 onward, Result-based convenience APIs (`json.get_field : (Json, String) -> Result[String, DecodeError]` etc.) will be added.

### Path-based local operations

```almide
type JsonPath =
  | Root
  | Field(parent: JsonPath, name: String)
  | Index(parent: JsonPath, i: Int)

json.get_path    : (Json, JsonPath) -> Option[Json]
json.set_path    : (Json, JsonPath, Json) -> Result[Json, PathError]  // strict: path must exist
json.upsert_path : (Json, JsonPath, Json) -> Json                     // create intermediates if missing
json.remove_path : (Json, JsonPath) -> Json                           // no-op if path missing
json.update_path : (Json, JsonPath, Fn[Json] -> Json) -> Result[Json, PathError]
```

Path literal syntax like `$.foo[0].bar` is deferred as future sugar; Phase 1 provides ADT-based JsonPath.

Having LLMs write operation sequences is more stable than full regeneration:

```almide
// ❌ Having LLM rewrite everything → risk of deleting existing fields
let new_config = json.parse(r#"{"host": "new", "port": 3000, ...}"#)?

// ✅ Local operations → other fields preserved (direct calls)
let new_config = json.upsert_path(config, Field(Root, "host"), json.s("new"))
let new_config = json.upsert_path(new_config, Field(Root, "port"), json.i(3000))

// ✅ pipe style (inserts as first argument)
let new_config = config
  |> json.upsert_path(Field(Root, "host"), json.s("new"))
  |> json.upsert_path(Field(Root, "port"), json.i(3000))

// set_path is strict: error if path doesn't exist
let result = json.set_path(config, Field(Root, "host"), json.s("new"))?
```

### Design decisions

- `json.s/i/f/b` replace `from_*`. 7 tokens to remember
- `json.object(List[(String, Json)])` means nested structure = code structure
- Existing `from_string`, `from_int`, etc. are deprecated
- No JSON literal syntax — typed records serve as literals

## Phase 2: deriving Codec

### Basics

```almide
type Person = { name: String, age: Int } deriving Codec

let j = json.encode(alice)                    // Person -> Json
let text = json.encode_to_string(alice)       // Person -> String

let bob = json.decode[Person](j)?             // Json -> Person
let bob2 = json.decode_from_string[Person](text)?  // String -> Person
```

### Derive codegen generation rules

| Target | encode | decode |
|--------|--------|--------|
| record | Output fields to object in declaration order | Look up by field name; missing follows default/Option rules |
| variant (Tagged) | `{"VariantName": {fields...}}` | Object assumed to have exactly one key. That key uniquely determines the variant (JSON objects are unordered, so determined by "sole key" not "first key") |
| variant (Adjacent) | `{"type": "Name", "data": {fields...}}` | Variant determined by tag field |
| variant decode failure | — | `InvalidValue` error (not `UnknownField`) |

### Integration with field defaults

```almide
type Config = {
  host: String = "localhost",
  port: Int = 8080,
  debug: Bool = false
} deriving Codec

// {"host": "example.com"} → Config { host: "example.com", port: 8080, debug: false }
// missing fields are filled with default values
```

### naming strategy

```almide
type ApiResponse = { userId: String, createdAt: String }
  deriving Codec(field_names: snake_case)

// encode: {"user_id": "...", "created_at": "..."}
// decode: {"user_id": "..."} → ApiResponse { userId: "..." }
```

Supported strategies:
- `identity` (default) — field name as-is
- `snake_case` — camelCase → snake_case
- `camel_case` — snake_case → camelCase

### ADT (variant type) representation

```almide
type Shape =
  | Circle(radius: Float)
  | Rect(w: Float, h: Float)
  deriving Codec

// Default: Tagged (safe, explicit)
// {"Circle": {"radius": 5.0}}

// Opt-in Adjacent
// deriving Codec(variant: adjacent("type", "data"))
// {"type": "Circle", "data": {"radius": 5.0}}
```

Default is **Tagged** (externally tagged). Reason:
- decode is unambiguous (variant determined by single-key object. RFC 8259: JSON objects are unordered, so determined by "uniqueness" of key, not "order")
- Untagged is prone to ambiguity → opt-in only, with constraints

### Manual implementation (remaining 5% of cases)

Since there are no field annotations, individual field renaming or custom transformations are written manually.

Manual codec connection rules:
- `deriving Codec` and manual implementation are mutually exclusive — don't write manual for derived types
- Encode-only manual / decode-only manual is possible (`deriving Encodable` + manual decode)
- Manual codec functions connect to the protocol via naming convention: `{type}_to_json` / `{type}_from_json`
- Manual codec function lookup is limited to the same module; compile error if multiple candidates exist
- Will migrate to explicit impl when traits are introduced

```almide
type Weird = { name: String, kind: String }

fn weird_to_json(w: Weird) -> Json =
  json.object([("name", json.s(w.name)), ("type", json.s(w.kind))])

fn weird_from_json(j: Json) -> Result[Weird, DecodeError] = {
  let name = json.get(j, "name") |> json.as_string
  let kind = json.get(j, "type") |> json.as_string
  match (name, kind) {
    (some(n), some(k)) => ok(Weird { name: n, kind: k }),
    _ => err(DecodeError { path: [], kind: MissingField("name or type") })
  }
}
```

## Phase 3: JsonOptions

```almide
type JsonOptions = {
  unknown_fields: UnknownFieldPolicy = Reject,
  trailing_commas: Bool = false,
  comments: Bool = false,
  pretty: Bool = false
}

type UnknownFieldPolicy = Reject | Ignore

// Usage
let config = json.decode_from_string[Config](text, JsonOptions { unknown_fields: Ignore })?
```

Default is `Reject` (reject unknown fields). Reason:
- Fail safe
- Catch typos and schema mismatches early
- Use explicit `Ignore` when you want to be lenient

`JsonOptions` controls parser/decode behavior, while `repair` is a separate layer for handling incomplete input.
`unknown_fields: Ignore` is a strict decode option ("this type tolerates additional fields"),
while repair's `RemoveField` is a "fix broken input" operation. Different philosophies.

## Phase 4: DecodeError + Repair + Validate

### Structured errors

```almide
type DecodeError = { path: List[PathItem], kind: DecodeErrorKind }

type PathItem =
  | Field(name: String)
  | Index(i: Int)
  | Variant(name: String)

type DecodeErrorKind =
  | SyntaxError(msg: String)
  | TypeMismatch(expected: String, got: String)
  | MissingField(name: String)
  | UnknownField(name: String)
  | InvalidValue(msg: String)
  | OutOfRange(msg: String)
  | DuplicateKey(key: String)
  | Custom(msg: String)
```

Display example: `error at .users[3].name: expected String but got Int`

### Validate (enumerate issues without decoding)

```almide
json.validate[T](value: Json) -> List[DecodeIssue]

type DecodeIssue = {
  path: List[PathItem],
  severity: Severity,
  kind: DecodeErrorKind,
  suggested_fix: Option[Fix]
}

type Severity = Error | Warning

type Fix =
  | CoerceType(from: String, to: String)    // "42" -> 42
  | FillDefault(field: String, value: Json)  // missing -> default
  | RemoveField(field: String)               // unknown field -> remove
  | WrapInArray                              // value -> [value]
```

Staged separation of parse / decode / validate:

```almide
// Step 1: Is it valid JSON?
let j = json.parse(text)?

// Step 2: Does it match the type? (strict)
let result = json.decode[Config](j)

// Step 2': Enumerate issues (without failing)
let issues = json.validate[Config](j)

// Step 2'': Decode while repairing
let repaired = json.repair[Config](j)
```

### Repair (differentiator for the LLM era)

```almide
json.repair[T](value: Json, level: RepairLevel = Safe) -> RepairResult[T]

type RepairLevel = Safe | Coercive

type RepairResult[T] =
  | Valid(value: T)
  | Repaired(value: T, fixes: List[Fix])
  | Invalid(issues: List[DecodeIssue])
```

Repair is not a replacement for decode, but an explicit opt-in auxiliary API. Normal decode always fails strictly.

Repair levels:
- **Safe** (default): Does not change known-field values; only fills in missing information or removes out-of-schema surplus information
- **Coercive**: Allows conversions that cross type boundaries. Explicit opt-in only

The boundary between Safe and Coercive is fixed in the spec. Ambiguous cases are not placed in Safe.

| Repair | Level | Reason |
|--------|-------|--------|
| missing field with default → fill with default value | Safe | Filling missing information, types are correct |
| missing optional field → fill with `none` | Safe | Natural completion per type definition |
| unknown field → ignore and drop | Safe | Removing out-of-schema surplus information. Known-field values unchanged |
| `"42"` → `42` (string → number coerce) | Coercive | Crosses type boundaries |
| single value → singleton array | Coercive | Structural conversion |
| Lossy numeric conversion (1.5 → 1) | Forbidden | Data loss (lossy) |
| Field value truncation | Forbidden | Data loss (lossy) |

```almide
// Safely accept incomplete JSON from LLM output
let input = json.parse(llm_output)?
match json.repair[Config](input) {
  Valid(config)             => use(config),
  Repaired(config, fixes)   => { log_fixes(fixes); use(config) },
  Invalid(issues)           => report_errors(issues),
}
```

No other language has this. Even Zod's `safeParse` is binary: pass or fail.
Repair is a third way — standardizing a **repairable boundary**.

### Schema Introspection

```almide
json.describe[T]() -> Json

// json.describe[Person]() =>
// {
//   "type": "object",
//   "properties": {
//     "name": {"type": "string"},
//     "age": {"type": "integer"}
//   },
//   "required": ["name", "age"]
// }
```

`json.describe[T]()` returns a **JSON Schema Draft 2020-12 compatible** machine-readable schema.
Usable for passing schemas in LLM tool calls (function calling).
By aligning with JSON Schema, it works directly with external tool integrations like OpenAI Structured Outputs.

Compliance:
- **Draft**: JSON Schema Draft 2020-12 (`$schema: "https://json-schema.org/draft/2020-12/schema"`)
- **Profile**: Only validation vocabulary guaranteed (format-annotation is optional)
- **`additionalProperties`**: Schemas generated by `deriving Codec` include `additionalProperties: false` by default (consistent with unknown field rejection)
- **nullable**: `Option[T]` is expressed as `{"oneOf": [{"type": "null"}, T_schema]}`
- **ADT (Tagged)**: Expressed as `{"oneOf": [{"type": "object", "properties": {"Circle": ...}, "required": ["Circle"], "additionalProperties": false}, ...]}`

Output format is treated as a stable API; breaking changes only in major versions.

Almide-specific type information (variant repr, default values, naming strategy) is expressed via JSON Schema extension properties (`x-almide-*`), maintaining compatibility with standard JSON Schema tools.

## Phase 5: Other formats

The same `deriving Codec` supports formats beyond JSON:

```almide
let bytes = msgpack.encode(alice)
let config = yaml.decode_from_string[Config](text)?
let data = cbor.decode[Packet](bytes)?
```

Since the Codec protocol is format-agnostic, each format only needs to implement an Encoder/Decoder.

## Key Design Decisions

### missing vs null

- **Field defaults take highest priority**: When a field default exists, apply the default for missing fields first. Only when no default exists, treat Option[T] as none
- `Option[T]` fields: both missing and null are treated as `none` (merge policy)
- Non-Option fields: missing → use default value if available, otherwise `MissingField` error
- null on non-Option → `TypeMismatch` error
- If strict distinction between missing/null is needed: consider `Presence[T] = Missing | Null | Value(T)` type in the future

### Numeric policy

- `1.0` on `Int` field → `TypeMismatch` (strict)
- `1` on `Float` field → allowed (Int promotes to Float)
- overflow → `OutOfRange` error
- NaN / Infinity → forbidden in encode (JSON spec compliant)
- In repair mode, `"42"` → `42` coercion is allowed

### Option during encode

- `none` → omit the field entirely
- `some(x)` → output the value
- Future: `deriving Codec(null_for_none: true)` to switch `none` → `null` output

### Circular references

- Not supported. Stack overflow during encode → error
- Almide is immutable-first, so circular structures are rare

### generic deriving

```almide
type Box[T] = { value: T } deriving Codec
// Only valid when T satisfies Codec
// Generates code that calls T's encode/decode at codegen time
```

Monomorphization-based (same approach as Rust codegen).

### field name collision

When using `deriving Codec(field_names: snake_case)`, compile error if converted keys collide:

```almide
type Bad = { userId: String, user_id: String }
  deriving Codec(field_names: snake_case)
// compile error: field name collision after snake_case conversion: "user_id"
```

### JSON literal syntax

Not included. Typed records serve as literals.
Having LLMs write JSON strings directly increases failure rate, so it is not the primary path.

### bytes / date / uuid

Representation policy for standard types not native to JSON:
- `bytes` → base64 string (auto-converted during encode/decode)
- `date/time` → RFC 3339 string
- `uuid` → canonical string
- `Map[NonString, V]` → encode as `List[(K, V)]`

## Implementation Order

1. **Phase 1** — Add `json.object`, `json.s/i/f/b`, path operation API to stdlib (supporting foundation)
2. **Phase 2** — Implement `deriving Codec` codegen — **the main target**. record → JSON, JSON → record. This is the heart of Almide's Codec experience
3. **Phase 3** — `JsonOptions` (unknown_fields, trailing_commas)
4. **Phase 4** — Structured `DecodeError` + `validate` + `repair` API + `json.describe[T]()` schema introspection
5. **Phase 5** — Add formats such as msgpack/yaml

Phase 1 is a supporting foundation for Phase 2; `deriving Codec` in Phase 2 is the core of the language experience.
In initial development, Phase 1-2 are tightly coupled; Phase 2 completion is the first milestone.
